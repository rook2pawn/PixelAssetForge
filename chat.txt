Yes. As of the newer Comfy ecosystem, there are basically **two paths**:

1. **Old / Manager path:** submit a PR to `ComfyUI-Manager`’s `custom-node-list.json`.
2. **New / preferred path:** publish to the **Comfy Registry**, which now powers install/discovery in current ComfyUI Manager. The official docs say every custom node published to the registry needs a `publisher` identifier in `pyproject.toml`. ([ComfyUI Documentation][1])

For a new suite, I would target the **Comfy Registry first**.

## Practical publish flow

### 1. Make the GitHub repo installable as a normal Comfy custom node

Typical structure:

```text
ComfyUI-AssetForge/
  __init__.py
  pyproject.toml
  requirements.txt
  LICENSE
  README.md
  nodes/
    color_masks.py
    mask_logic.py
    image_filters.py
  examples/
    semantic_color_map_workflow.json
```

Your `__init__.py` should expose the usual Comfy mappings:

```python
from .nodes.color_masks import NODE_CLASS_MAPPINGS, NODE_DISPLAY_NAME_MAPPINGS

__all__ = ["NODE_CLASS_MAPPINGS", "NODE_DISPLAY_NAME_MAPPINGS"]
```

The exact internal structure can vary, but from Comfy’s perspective the important thing is that importing the repo gives it `NODE_CLASS_MAPPINGS`.

---

### 2. Add proper registry metadata

The official flow uses `pyproject.toml`. Their docs show this shape:

```toml
[project]
name = "comfyui-assetforge"
description = "Deterministic asset-generation utility nodes for ComfyUI."
version = "0.1.0"
license = { file = "LICENSE" }
dependencies = []

[project.urls]
Repository = "https://github.com/YOUR_NAME/ComfyUI-AssetForge"

[tool.comfy]
PublisherId = "your-publisher-id"
DisplayName = "ComfyUI AssetForge"
Icon = "https://raw.githubusercontent.com/YOUR_NAME/ComfyUI-AssetForge/main/assets/icon.png"
```

The registry docs also recommend using `comfy node init` to generate the metadata scaffold. ([ComfyUI Documentation][1])

---

### 3. Create a Comfy Registry publisher

Go to the Comfy Registry, create a publisher identity, and copy the publisher id into `[tool.comfy].PublisherId`. The docs note that the publisher id is globally unique and cannot be changed later. ([ComfyUI Documentation][1])

So choose something you can live with.

---

### 4. Create a publishing API key

In the Registry, create a publishing API key for that publisher. The docs say this key is used to publish custom nodes to the Registry / ComfyUI Manager via CLI or GitHub Actions. ([ComfyUI Documentation][1])

Do **not** commit it. Use it locally or as a GitHub Actions secret.

---

### 5. Publish with the Comfy CLI

The official docs give the manual publish command as:

```bash
comfy node publish
```

It prompts for the API key, packages the git-tracked repo, and publishes the semver version. ([ComfyUI Documentation][1])

I’d do a manual publish first before automating CI.

---

### 6. Add `.comfyignore`

The registry package includes git-tracked files by default, so add `.comfyignore` for things you do not want shipped:

```gitignore
docs/dev/
tests/
.vscode/
*.psd
*.kra
*.clip
*.blend
examples/large/
```

The official docs say `.comfyignore` uses `.gitignore`-style syntax and is meant to exclude dev-only files from the published archive. ([ComfyUI Documentation][1])

---

## What about ComfyUI-Manager?

The Manager GitHub still documents the older path: add an entry to `custom-node-list.json` and submit a PR. It also points custom-node developers toward the newer Registry docs. ([GitHub][2])

So I would think of it like this:

```text
Modern path:
  Publish to Comfy Registry

Legacy/extra visibility path:
  PR to ComfyUI-Manager custom-node-list.json if needed
```

The Comfy blog says the Registry is meant to improve stability, security, reproducibility, and discoverability, and newer Manager versions install from the Registry. ([ComfyUI Blog][3])
